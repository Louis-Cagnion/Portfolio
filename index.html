Mes projets d'école

Durant mon cursus, j'ai eu l'occasion de réaliser plusieurs projets solo et de groupe particulièrement intéressants.

Push_swap

Le but de ce projet était de coder un algorithme de tri le plus efficace possible afin de trier jusqu'à 500 nombres.

Pour obtenir la note maximale, il fallait réussir à trier une liste aléatoire de 500 entiers en moins de 5500 opérations avec des mouvements très limités.

En résumé, imaginez deux listes : la seconde est vide, et vous ne pouvez interagir qu'avec le premier ou le dernier élément de chaque liste afin de déplacer les nombres.

J'avais réussi à développer un algorithme qui effectuait le tri en environ 5700 coups en moyenne. J'étais fier de cette méthode trouvée après avoir testé plusieurs approches différentes, même si j'ai finalement dû m'inspirer d'un algorithme existant pour passer sous la barre des 5500 opérations.

Le bonus consistait à créer un "Checker" permettant de vérifier qu'une suite d'opérations donnait bien, à la fin, une liste triée dans l'ordre croissant.

Fract-ol

Ce projet était plus artistique. Le but était d'utiliser la MinilibX afin de créer un rendu graphique de fractales.

Pour ceux qui ne savent pas ce qu'est une fractale : il s'agit d'un motif qui se répète à l'infini lorsqu'on zoome dessus. Le chou romanesco est d'ailleurs un excellent exemple de fractale naturelle.

Les bonus consistaient à :

  • Créer une troisième fractale en plus de Julia et Mandelbrot
  • Faire suivre le zoom avec la position de la souris
  • Déplacer la caméra avec les flèches directionnelles
  • Modifier dynamiquement les couleurs de la fractale

Minishell

Le but de ce premier projet de groupe était de recréer un mini shell.

Il fallait notamment :

  • Pouvoir naviguer dans différents niveaux du shell
  • Recréer plusieurs commandes Bash comme cd, echo, pwd, export ou exit
  • Gérer l'historique
  • Implémenter les redirections (<, >, <<, >>)
  • Gérer les variables d'environnement
  • Gérer les signaux clavier comme Ctrl+C et Ctrl+D

Les bonus consistaient à ajouter les opérateurs logiques '&&' et '||', ainsi qu'une gestion des priorités avec des parenthèses.

C'est moi qui me suis occupé de cette partie. Cela m'a permis de travailler sur des arbres binaires et toute la logique d'exécution liée aux priorités des commandes.

Philosophers

Pour ce projet, il fallait réussir à faire survivre des philosophes.

Ces philosophes possédaient plusieurs états :

  • Un temps pour manger
  • Un temps pour réfléchir
  • Un temps pour dormir
  • Un nombre de repas à atteindre (optionnel)

Ils possédaient également un temps de vie limité. Le but était de fournir différents paramètres de temps et différents nombres de philosophes, puis d'être suffisamment précis pour réussir à les maintenir en vie selon les contraintes données.

Les temps étaient exprimés en millisecondes, donc le moindre mauvais calibrage se remarquait immédiatement.

En plus de cela, il fallait afficher précisément le moment auquel chaque philosophe effectuait une action, avec une marge d'erreur inférieure à 10 ms.

La particularité de ce projet venait du fait que les philosophes ne pouvaient manger que si deux fourchettes étaient disponibles. Cependant, chaque philosophe ne possédait qu'une seule fourchette. Il fallait donc organiser correctement les accès afin d'éviter les blocages, les conflits et les morts prématurées.

Les bonus consistaient à refaire le même exercice en remplaçant les threads par des processus enfants et les mutex par des sémaphores.

Les bonus consistaient notamment à :

  • Créer un site web statique hors PHP

Webserv

L'avant-dernier projet de groupe du tronc commun consistait à réaliser notre propre serveur HTTP.

Le serveur devait notamment :

  • Être configuré avec un fichier de configuration créé de zéro
  • Être non bloquant
  • Utiliser un seul appel à poll
  • Faire en sorte que tous les read et write soient exécutés après poll
  • Éviter l'utilisation directe de errno après un read ou write
  • Gérer les boucles infinies pour éviter de bloquer le serveur
  • Fonctionner correctement sur un navigateur
  • Avoir des pages d'erreurs
  • Retourner les bons codes HTTP
  • Utiliser des CGI
  • Gérer les méthodes GET, POST et DELETE
  • Permettre l'upload de fichiers
  • Écouter sur plusieurs ports avec différents contenus

C'était déjà un énorme morceau pour la partie obligatoire.

Pour les bonus, il fallait gérer les sessions des clients et les cookies, ainsi qu'ajouter plusieurs types de CGI.

Transcendence

Pour ce dernier projet du tronc commun, qui était également un projet de groupe, le but était de créer une véritable application web complète.

Le projet devait contenir :

  • Un frontend
  • Un backend
  • Une base de données

Comme nous étions nombreux à travailler dessus, il fallait également garder une organisation claire avec des commits propres et compréhensibles.

L'application devait pouvoir être lancée via une seule commande à l'intérieur d'un conteneur, fonctionner correctement sur navigateur, ne produire aucune erreur dans la console, et inclure une politique de confidentialité ainsi que des conditions d'utilisation.

Concernant la stack technique, il fallait notamment :

  • Créer un frontend clair, réactif et responsive
  • Utiliser un framework CSS
  • Stocker les informations sensibles dans un fichier .env ignoré par Git
  • Créer une base de données bien structurée
  • Mettre en place une authentification sécurisée
  • Valider les données côté frontend et backend
  • Utiliser HTTPS côté backend

Ensuite, nous pouvions choisir différents modules à ajouter au projet.

Certains modules étaient mineurs et rapportaient un point, tandis que d'autres étaient majeurs et rapportaient deux points.

Le but était de totaliser 14 points pour valider la partie obligatoire, puis 19 points pour valider les bonus.

Pour ma part, je me suis occupé de deux modules majeurs :

  1. Créer une IA capable d'affronter les joueurs
  2. Créer une version 3D du jeu
